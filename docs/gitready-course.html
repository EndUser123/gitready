02

What happens when you run it

Here's the exact journey your folder takes from "bare project" to "GitHub-ready portfolio."

When you run /gitready my-project, gitready walks through 7 phases — automatically, in order:

1
DETECT
Scans your folder. Finds what's there, what's missing.
2
ANALYZE
Is it a plugin? A skill? A Python library? Determines the right type.
3
GENERATE
Creates scaffold, badges, CI/CD, CHANGELOG, docs — whatever's missing.
4
VALIDATE
Runs quality checks — security scanning, dependency audit, badge validation.
5
CLEANUP
Removes obsolete files left over from refactoring.
6
PUBLISH
Extracts from monorepo, creates GitHub repo, pushes code.
7
REPORT
Shows everything that was created, with evidence.
ℹ️

No mode selection required. gitready detects what's needed and runs the right phases automatically. You just type one command.

What gets created

Every run produces a specific set of files. Here's the complete map.

# This is what gitready creates — pick what you need your-package/ ├── .claude-plugin/ │ └── plugin.json ← Name, description, author (the plugin manifest) │ ├── .github/workflows/ │ └── test.yml ← Runs tests on every push (CI/CD) │ ├── core/ │ ├── __init__.py ← Package marker │ └── main.py ← Your Python code │ ├── hooks/ │ └── hooks.json ← Hook configuration (plugins only) │ ├── skills/ │ └── your-skill/ │ └── SKILL.md ← Skill definition (skills only) │ ├── scripts/ │ ├── extract_from_monorepo.py ← GitHub publication │ ├── create_github_repo.py ← Repo creation │ ├── finalize_github_repo.py ← GitHub Pages, releases │ └── scan_package_quality.py ← Security & quality scan │ ├── docs/ │ └── diagrams/ │ └── c4_*.mmd ← Architecture diagrams │ ├── assets/ │ ├── banners/ ← Social media banners │ ├── videos/ ← NotebookLM explainer video │ └── slides/ ← Presentation deck │ ├── README.md ← Portfolio-ready with badges & flowchart ├── LICENSE ← MIT (or your choice) ├── CHANGELOG.md ← Auto-generated version history ├── AGENTS.md ← AI-maintainable documentation └── CONTRIBUTING.md ← Contribution guidelines
💡

Not all files are created for every package type. A skill gets skills/your-skill/SKILL.md but no core/. A library gets a test.yml but no hooks.json. gitready creates only what's appropriate.

05

What the badges actually mean

Every shield on a README has a specific job. Here's what each one communicates.

What you see
Version — shields.io/badge/version-5.5.0-blue
License — shields.io/badge/license-MIT-green
Tests — shields.io/badge/tests-passing-brightgreen
What it actually says

"This project is at version 5.5.0. You can pin to this release if you want stability."

"This project uses the MIT license — use it freely, no strings attached."

"Automated tests run on every change and they're all passing right now."

💡

Why it matters: When a recruiter or open-source maintainer lands on your README, badges are the 3-second gut check. Passing tests + current version = "this project is maintained." No badges = "this might be abandoned."

The badge lifecycle

1
CI/CD runs test.yml
Every push triggers GitHub Actions. Results go to the test badge URL.
2
shields.io fetches the status
The badge image is generated dynamically from the CI/CD result URL.
3
README shows live status
The badge refreshes every time someone loads the page — it's always current.
06

The CI/CD pipeline in plain English

Here's what actually happens when GitHub Actions runs your test.yml.

CODE 
name: Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
      - run: pip install pytest
      - run: pytest
PLAIN ENGLISH

"This workflow is called 'Tests'"

"Run this every time someone pushes code or opens a pull request"

"Define a job called 'test' that runs on a fresh Ubuntu machine"

"Step 1: Download your code from GitHub"

"Step 2: Set up Python on that machine"

"Step 3: Install pytest (the test runner)"

"Step 4: Run the tests"

✅

You write code and push. GitHub automatically runs your tests on a clean machine. You don't have to remember to run them — GitHub handles it.

08

Three things that trip people up

Each one has a clear recovery path — here's what to watch for and how to fix it.

🔌
Wrong package type selected

A plugin and a Python library have completely different structures — not just different folders, but different purposes. If you pick the wrong one, commands won't register where you expect.

How to recover: Run /gitready --dry-run to preview what will be created before committing. The decision tree in Module 4 shows which type to pick.

🔗
Broken symlinks after brownfield conversion

After converting an existing library, hook symlinks in P:/.claude/hooks/ sometimes still point to the old src/ path — which no longer exists. Your hooks silently stop working.

How to recover: gitready's output tells you to verify symlinks. Don't skip that step. Run gitready --target your-package again to let it fix the paths automatically.

📦
Using /plugin during local development

The /plugin install command copies your package to the plugins cache — it doesn't link back to your source. Every change requires a reinstall. That's fine for distributing to others, but painful when you're iterating.

How to recover: For active development, use a junction (Windows, no admin needed) or symlink (macOS/Linux) pointing directly at your source. gitready's output shows the right command for your OS.

💡

Windows note: Use junctions for directories (no admin required). Use symlinks for individual files (requires Developer Mode or admin). gitready always shows the correct approach for your OS.
